# Housekeeping -- Codebase Cleanup Sweep

Periodic maintenance skill that removes accumulated cruft (build artifacts, temp files, stale reports), detects potentially unused `.dart` files, and sweeps stale imports or phantom routes. Runs in dry-run mode by default -- always shows what it will do before doing it.

References:
- `dev/conductor/journal/INDEX.md` (journal tag index)
- `dev/audits/FIX_MANIFEST.md` (active bug tracker -- never delete)
- `dev/conductor/Skills/README.md` (skill index)

---

## Safety Rules (NON-NEGOTIABLE)

1. **Never auto-delete `.dart` files** -- only report candidates for human review.
2. **Never delete active mission docs, standards, skills, or their `README.md`/`INDEX.md` files.**
3. **Never delete `FIX_MANIFEST.md`, `THOUGHTS.md`, `HANDOFF.md`, `ACTIVE_CONTEXT.md`, `CONDUCTOR_HEALTH.md`.**
4. **Git-tracked files require user confirmation** before removal.
5. **Prototype directories require explicit user confirmation** per directory.
6. **Default is dry-run** -- show the plan, get a "go" before executing.

---

## Step 0: Dry-Run Scan

Before any deletion or patching, run all 7 passes in **scan mode**. Collect findings into a summary table and present it to the user.

```
HOUSEKEEPING SCAN RESULTS
=========================
Pass 1 (Artifacts & Temp):  N files, ~X MB reclaimable
Pass 2 (Prototypes):        N dirs flagged, ~X MB reclaimable
Pass 3 (Journal):           N entries older than 90 days
Pass 4 (Reports):           N reports flagged for archival
Pass 5 (Dead Dart):         N candidate files detected
Pass 6 (Imports/Routes):    N stale imports, N phantom route sites
Pass 7 (Summary):           Total ~X MB reclaimable

Proceed with cleanup? [specify which passes to run, or "all"]
```

Only proceed after user confirms which passes to execute.

## Pass 1: Build Artifacts & Temp Files (Safe -- Always Run)

These are regenerable files that waste disk space and pollute git status.

### Targets

| Pattern | Location | Why Safe |
|---------|----------|----------|
| `.dart_tool/` dirs | `dev/design/prototypes/*/` | Regenerated by `flutter pub get` |
| `.DS_Store` | Repo-wide | macOS metadata, never needed |
| `*.log` files | `dev/` (recursive) | Debug logs, not source |
| `*.tmp` files | `dev/` (recursive) | Temporary artifacts |
| `pubspec.lock` | `dev/design/prototypes/*/` | Regenerated, not the main app lock |
| `build/` dirs | `dev/design/prototypes/*/` | Compile output |
| Chrome device data | `dev/design/prototypes/*/.dart_tool/chrome-device/` | Browser session cache |

### Commands

```bash
# Scan
find dev/design/prototypes -name '.dart_tool' -type d 2>/dev/null
find . -name '.DS_Store' -not -path './.git/*' 2>/dev/null
find dev/ -name '*.log' -o -name '*.tmp' 2>/dev/null
find dev/design/prototypes -name 'build' -type d 2>/dev/null
find dev/design/prototypes -name 'pubspec.lock' 2>/dev/null

# Execute (after confirmation)
find dev/design/prototypes -name '.dart_tool' -type d -exec rm -rf {} +
find . -name '.DS_Store' -not -path './.git/*' -delete
find dev/ \( -name '*.log' -o -name '*.tmp' \) -delete
find dev/design/prototypes -name 'build' -type d -exec rm -rf {} +
find dev/design/prototypes -name 'pubspec.lock' -delete
```

### Verify

```bash
# Confirm deletion
du -sh dev/design/prototypes/
```

## Pass 2: Prototype Pruning (Interactive -- Confirm Per Directory)

Prototype directories in `dev/design/prototypes/` are full Flutter projects that can be 200+ MB each. Once exported to production, they're dead weight.

### Methodology

For each directory in `dev/design/prototypes/`:

1. **Identify key widgets/classes** in the prototype:
   ```bash
   grep -rn 'class ' dev/design/prototypes/<name>/lib/ | head -20
   ```

2. **Check if exported** -- search for those classes in `lib/`:
   ```bash
   grep -rn 'ClassName' lib/
   ```

3. **Check age** -- last modified date:
   ```bash
   find dev/design/prototypes/<name> -name '*.dart' -exec stat -f '%m %N' {} + | sort -rn | head -1
   ```

4. **Check for MISSION.md** -- if it says "exported" or "complete", it's a candidate.

### Decision Matrix

| Exported to `lib/`? | Age > 30 days? | Action |
|---------------------|----------------|--------|
| Yes | Yes | Flag for removal (confirm with user) |
| Yes | No | Flag but note "recently exported" |
| No | Yes | Skip -- may still be in progress |
| No | No | Skip -- active prototype |

### Execute

Only after user confirms each directory:
```bash
rm -rf dev/design/prototypes/<name>/
```

Update any references in `dev/design/` docs that point to the removed prototype.

## Pass 3: Journal Compaction (Interactive)

Journal entries capture session learnings. Over time, their insights get absorbed into standards and ground rules, making the original entries redundant.

### Methodology

1. **List entries older than 90 days**:
   ```bash
   find dev/conductor/journal -name '*.md' -not -name 'INDEX.md' -not -name 'README.md' -mtime +90
   ```

2. **For each old entry**, read and check:
   - Are its key findings already in a standard or ground rule?
   - Does `INDEX.md` reference it? (tag counts will need updating if removed)
   - Is it the only source for a unique insight?

3. **Classification**:

   | Status | Action |
   |--------|--------|
   | Fully absorbed into standards | Safe to remove (update INDEX.md tag counts) |
   | Partially absorbed | Keep -- note which insight is missing from standards |
   | Unique insight, not in any standard | Keep -- consider promoting to a standard |

### Protected Files (Never Delete)

- `dev/conductor/journal/INDEX.md`
- `dev/conductor/journal/README.md`

### Execute

For each confirmed removal:
1. Delete the journal file
2. Update `INDEX.md` -- decrement tag counts, remove filename references
3. If a tag count drops to 0, remove the tag entry

## Pass 5: Dead Dart File Detection (Read-Only Scan)

Find `.dart` files in `lib/` that no other file imports or references. These are candidates for removal but **must be reviewed by a human**.

### Methodology

1. **Build the file list** (exclude generated and known entry points):
   ```bash
   find lib/ -name '*.dart' \
     -not -name '*.g.dart' \
     -not -name '*.freezed.dart' \
     -not -name 'main.dart' \
     -not -name 'firebase_options.dart' \
     -not -path '*/generated/*'
   ```

2. **For each file, check if it's imported anywhere**:
   ```bash
   # Extract the import path from the filename
   # e.g., lib/screens/foo/bar.dart -> package:indep_app/screens/foo/bar.dart
   # Also check relative imports: import 'bar.dart' or import '../foo/bar.dart'

   basename="bar.dart"
   grep -rn "$basename" lib/ --include='*.dart' | grep -v "^lib/screens/foo/bar.dart"
   ```

3. **Exclude known non-imported but valid files**:
   - Route module files (referenced in `app_router.dart` by path, not import)
   - Barrel/index files (`models.dart`, `states.dart`, `effects.dart`)
   - Seed data files (referenced at runtime, not imported statically)
   - Test files
   - Files registered in `pubspec.yaml` or `build.yaml`

4. **Cross-reference with `flutter analyze`**:
   ```bash
   flutter analyze 2>&1 | grep 'unused_import'
   ```

### Output Format

```
DEAD DART FILE CANDIDATES
=========================
LIKELY DEAD (0 imports found):
  - lib/screens/old_feature/old_screen.dart
  - lib/widgets/deprecated_card.dart

POSSIBLY DEAD (only self-imports or test imports):
  - lib/services/unused_service.dart

FALSE POSITIVE RISKS (barrel-exported or route-referenced):
  - lib/screens/foo/foo_screen.dart  (check route config)

ACTION: Review each candidate manually before deleting.
         Run `grep -rn "ClassName" lib/` to verify.
```

**Never delete any file in this pass.** Present the report and let the user decide.

## Pass 6: Stale Imports & Phantom Route Sweep (Read-Only First)

Find analyzer-backed stale imports and navigation calls that target missing or stale GoRouter paths. This pass is read-only by default; only patch issues that are explicitly confirmed.

### Methodology

1. **Capture analyzer-backed import findings**:
   ```bash
   flutter analyze 2>&1 | grep 'unused_import'
   ```

2. **Build the route map from the router**:
   ```bash
   rg -n "GoRoute\\(|path:\\s*'|name:\\s*RouteNames" lib/routing/app_router.dart lib/routing/modules -g '*.dart'
   ```

3. **Inventory literal navigation call sites**:
   ```bash
   rg -n "context\\.(go|push|replace)\\('/" lib/ -g '*.dart'
   rg -n "context\\.canPop\\(\\) \\? context\\.pop\\(\\) : context\\.go\\('/'\\)" lib/ -g '*.dart'
   ```

4. **Classify each finding**:

   | Finding Type | Safe to Patch? | Rule |
   |-------------|----------------|------|
   | Analyzer `unused_import` | Yes | Remove only the exact import the analyzer proves redundant |
   | Literal route to missing path (example: `context.go('/')` with no `/` route) | Yes | Replace only after verifying the canonical destination in `app_router.dart` |
   | Dynamic/interpolated route string | No | Review manually; do not rewrite blindly |
   | Path inside an orphaned screen/file | No | Note it, but do not patch unless the user wants dormant code cleanup too |
   | Named-route or redirect semantics issue | Review first | Confirm `go` vs `push` intent before changing behavior |

5. **Patch only confirmed cases, then re-run analyzer**:
   ```bash
   flutter analyze
   ```

### Output Format

```
STALE IMPORT / ROUTE FINDINGS
============================
SAFE TO PATCH:
  - lib/foo.dart:12  unnecessary import flagged by analyzer
  - lib/bar.dart:88  context.go('/') but router only registers /hub

REVIEW FIRST:
  - lib/baz.dart:42  interpolated route string; verify target before editing
  - lib/qux.dart:19  orphaned screen uses stale path; dormant until screen is revived

ACTION:
- Patch only the SAFE TO PATCH items
- Re-run flutter analyze
- Report any REVIEW FIRST items without rewriting them
```

### Anti-Drift Notes

- Treat `/` as invalid unless `app_router.dart` explicitly registers `/`
- Prefer route-module truth over screen assumptions
- Do not convert `push` to `go` or vice versa unless the stack behavior is clearly wrong
- A stale import fix is safe only when backed by analyzer output, not just grep

## When to Run

| Trigger | Recommended Passes |
|---------|-------------------|
| Monthly maintenance | All passes |
| Before a release | Pass 1 + Pass 5 + Pass 6 |
| After completing a mission | Pass 2 + Pass 3 |
| After a sprint | Pass 4 |
| Disk space pressure | Pass 1 + Pass 2 |
| After router edits | Pass 6 |
| Code review / refactor | Pass 5 + Pass 6 |

---

## Anti-Patterns

| Anti-Pattern | Why It's Wrong | Do This Instead |
|-------------|---------------|----------------|
| Auto-delete `.dart` files | Could remove active code | Report only, human decides |
| Delete prototype without checking export status | Loses unreplicated design work | Always check `lib/` for exported classes first |
| Delete journal entries without checking standards | Loses unique insights | Verify insight is captured elsewhere first |
| Skip the dry-run scan | User can't review what's being deleted | Always show the plan first |
| Blindly rewrite route strings | Can break stack semantics or hide real router drift | Compare against `app_router.dart` + route modules first |
| Remove imports just because grep says they look redundant | Can break type resolution or exports | Only remove imports explicitly flagged by `flutter analyze` |
| Delete `FIX_MANIFEST.md` or `SPRINT.md` | Destroys active tracking | These are permanently protected |
| Run during active sprint | Could delete in-progress claims | Wait until sprint is archived |
